Royal Ambush

  • Originality: What is original about your game?

    • The rules of the games are based on an original analogue prototype (described in the short attached document). Only the first phase of those rules are implemented in this prototype. It is also a 2 player game, requiring players to share the mouse and trade and their turn comes around.

  • Technical Merit: What is interesting about your game technology?

    • Hex coordinate system, sleek animations and transitions, responsive kinaesthetic, tile generation and population (could be randomized if the ruleset required to it). dynamic tile-based user functionality, abstract proximity detection. Figuring out how to generate and interact with hex board was actually extremely involved for me, with a lot of manual calculations to make sense of hexagonal x,y,z and relative positioning and logic. Much of my effort was to make a solid foundation of code for any type of game that uses a regular hexagonal tile system -- not just this simple demonstration.

  • Prototype Postmortem: What did you learn from this prototype? What was the easiest or hardest part of making it?

    • Learning the principle design patterns in games the hard way. Thinks like phase loops and control pass-offs between different components. Encapsulating logic and learning the best practices was a big challenge as well. I really tried and immerse myself in learning the "right" way to code a game, rather than taking a bunch of shortcuts and making a one-off demo in a comfy, but non-expandable way. I look forward to implementing the "battle" phase when I get the chance to improve on this prototype. Then we can have an actual win condition and a proper closed loop.

  • Prototype Assets:

    • I made all of my graphical assets, using generic Photoshop assets and hand-drawings.
    Audio assets are used under Creative Commons by "OtoLogic"

  • Prototype Closest Other Game:

    • Really all the code is completely from scratch. It is written in typescript -- and I extended the classes which made most sense to me, as a new dev in Phaser. Technically you could say I started with the minimal template, but rewrote everything.

  • High Concept:

    • Fun modern twist on ancient traditional board game 'Tafl'

  • Theme:

    • Challenge: Must be played with someone else/Things are only added, not removed. (Concept was targeting DA 1 challenges but carried over due to delays.)

  • Mandated Variety:

    • Input: Mouse, clickable buttons; Randomness: little-to-none; Play-style: Competitor, craftsmen, storyteller; Genre: board game

  • Prototype Goal:

    • The goals really revolved around demonstrating: two players being able to interact with the same objects on a screen, in a controlled fashion with expected inputs being guided. Visually and logically transferring control among the players and the game loop; interactive hexagonal tile system based on 2.5d x,y,z coordinates. THe "look" and "feel" of a feature-complete game that fully implements the board game rules.
